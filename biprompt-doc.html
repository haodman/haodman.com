为什么需要它
System Prompt 调试的真实痛点

随着 Agent 能力不断升级，越来越多的项目开始接触超长/系统级 System Prompt 调试（如 AI Remix、AI 视觉日记等），这些场景中，以下问题几乎每天都在发生。

痛点 01

阅读效率极低，英文门槛高

Prompt 普遍以英文书写，中文背景的开发者阅读、理解、修改时摩擦极大，反复对照翻译严重打断思路。

痛点 02

来回复制粘贴，容易出错遗漏

依赖 AI 翻译时需要手动复制 → 粘贴到翻译工具 → 再复制回来，流程繁琐，内容极易在复制过程中出现错位或遗漏。

痛点 03

全量翻译耗时长、Token 消耗巨大

一份 3w 字的 System Prompt 每次全量重翻，不仅等待时间长，Token 成本也高得难以接受。改一个词，却要付出翻译整篇的代价。

痛点 04

Prompt 结构混乱，难以维护

长 Prompt 缺乏统一结构规范，Role/Rules/Output 混在一起，修改一个模块时容易影响其他部分，越改越乱。

痛点 05

双语内容难以精准定位

中英文版本各是一个长文档，修改后不知道对应位置在哪、有没有遗漏，需要大量手动比对，效率极低。

痛点 06

翻译质量无法实时校验

AI 翻译存在漏译、错译的风险，但当前工具很难做到实时对照校验，错误往往在最终测试时才被发现。

核心价值主张
针对以上痛点，BiPrompt 解决了——
超长 Prompt 难编辑、难翻译、高 Token 成本问题

使用教程
5 步上手 BiPrompt

从零开始，用 BiPrompt 完成一个完整的双语 System Prompt 编写流程。

1
创建 Prompt 项目
打开 BiPrompt，点击「新建 Prompt」，为你的 Prompt 命名（如 "AI Remix System Prompt"），选择基础语言对（默认中文 ↔ English）。
💡 如果已有一份现成的 Prompt，可以直接粘贴到任意一侧，BiPrompt 会自动识别结构并提示是否拆分为模块。
2
搭建模块结构
从预设模块中选择你需要的类型，或点击「+ 新增模块」自定义命名。建议优先搭好整体骨架再填充内容。
// 推荐的标准模块结构
📌 Role — 角色定义与身份设定
🌐 Context — 背景信息与任务说明
📋 Rules — 行为约束与边界条件
🛠️ Skill Pool — 具体技能与能力列表
📤 Output — 输出格式与结构要求
📚 Examples — 少样本示例（可选）
⚠️ 模块名称和类型标签不会进入最终编译结果，仅作为编辑器的结构辅助信息。
3
在任意语言侧编写内容
选择你更习惯的语言开始写作。在左侧（中文）或右侧（英文）的对应模块内直接输入内容，无需提前准备双语版本。
✨ BiPrompt 支持 Markdown 格式：使用 # ## ### 添加标题，- 或 1. 创建列表。
💡 写完某个模块后，点击模块右上角的「翻译」按钮，或等待自动延迟触发翻译。
4
使用局部翻译高效迭代
这是 BiPrompt 最核心的工作流程。修改任意一侧的某个模块时，只有该模块会触发重新翻译，其他模块保持不变。
⚡ 局部翻译 vs 全量翻译：修改一条 Rules，全量翻译需要处理整个 3w 字 Prompt；局部翻译只处理该模块的几十个字，Token 消耗降低约 90%。
🔄 鼠标悬停某一段文字时，对侧对应段落会自动高亮，方便实时比对翻译质量。
🖱️ 双侧内容自动同步滚动，无需手动调整两个滚动条的位置。
5
编译并导出完整 Prompt
内容编写完成后，点击顶部工具栏的「编译」按钮，BiPrompt 将所有启用模块按顺序拼接为完整 Prompt。
📋 三种复制模式： ① 仅复制中文版 ② 仅复制英文版 ③ 双语合并（中英对照格式）
📤 支持导出为 .md 或纯文本文件，方便归档或直接用于 API 调用。
🔢 编译预览页面会显示总 Token 数，便于在部署前评估成本。

1. Responses must be concise... ← Rules
2. Avoid jargon...
Output in the following format:... ← Output
